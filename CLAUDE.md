# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## About

Unified CLI for Atlassian **Confluence** and **Jira**, written in Rust (edition 2024, MSRV 1.94). Non-interactive by design, structured output, multiple profiles, and a generic `atl api` passthrough for endpoints without first-class support.

For a deeper tour and a step-by-step "add a new subcommand" walkthrough, see [`docs/project-layout.md`](docs/project-layout.md). User-facing docs are in `README.md`. The auto-generated CLI reference in `docs/reference/atl.md` is regenerated by CI and **must not be edited by hand** — see "Docs generation" below.

## Build & test

```bash
cargo check
cargo build
cargo build --release
cargo test                          # all tests
cargo test <substring>              # filter to matching tests
cargo test -- --nocapture           # see stdout in tests
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
```

Pre-commit hooks live in `lefthook.yml` (fmt-check, clippy, test). The deny rules in `.claude/settings.json` block bypassing them — never use `--no-verify` / `-n`. Hooks can take ~10s; that's normal.

`cargo test` runs both unit tests (in `src/`) and integration test binaries under `tests/`. The Atlassian contract tests (`tests/contract_*.rs`) are `#[ignore]` by default — they only run when a Prism mock server is up.

## Architecture (the parts that aren't obvious from `ls`)

### Thin `main.rs`, manual Tokio runtime

`src/main.rs` does five things in `run()`:

1. Pre-clap **alias expansion** via `commands::alias::expand_aliases(argv)` so user-defined aliases are rewritten *before* clap sees them.
2. Parse with `Cli::parse_from(args)`.
3. Construct `IoStreams::system(&cli)` once and thread it through dispatch.
4. Dispatch via the `Command` enum match. Sync commands (`init`, `config`, `completions`, `self`, `alias`, `generate-docs`) call directly; async commands wrap their handler in `run_async()` which spins up a `tokio::runtime::Builder::new_multi_thread()` on demand. There is **no `#[tokio::main]`** — sync commands don't pay the runtime cost.
5. After a successful dispatch, call `commands::updater_notifier::maybe_print_notice(io, is_self_cmd)` (best-effort; suppressed for `atl self` itself, in CI, or if `ATL_NO_UPDATE_NOTIFIER=1`).

### `IoStreams` everywhere — never `println!`

`src/io/{mod,pager}.rs` defines `IoStreams`, the only legitimate way for command code to write output. It owns:

- stdout/stderr/stdin handles (`Box<dyn Write/BufRead>`)
- TTY detection (`is_stdout_tty`, `is_stderr_tty`, `is_stdin_tty`)
- Color resolution (precedence: `--no-color` > `NO_COLOR` env > `CLICOLOR_FORCE` > TTY-detect)
- Pager management — `start_pager()` / `stop_pager()` spawn `$ATL_PAGER` > `$PAGER` > `less -FRX` (or `more` on Windows)

Every command handler takes `io: &mut IoStreams` as a parameter. The pager is engaged automatically by `confluence::run`/`jira::run` for long-output commands (`read`, `view`, `search`) **only when** format is `Console` AND stdout is a TTY AND `--no-pager` is not set. Outside the engagement window, `io.stdout()` is a normal stdout handle.

`println!`/`eprintln!` belong only in `main.rs` panic paths, `tracing::*!` macros, and the `init` wizard. Everything else uses `io.stdout()` / `io.stderr()`.

### Output: build a `Value`, hand it to `write_output`

Command handlers do **not** print directly. They build a `serde_json::Value` and call `crate::output::write_output(value, format, io, transforms)`, which:

1. Runs the value through `transform::apply` — applies the global `--jq` (via `jaq` 2.x) and `--template` (via `minijinja`) flags if set. `--template` switches output to text mode regardless of `--format`.
2. Picks a `Reporter` via `reporter_for_format(format, io.color_enabled())`. Reporters are in `src/output/{console,json,csv_out,toml_out,toon}.rs`.
3. Writes to `io.stdout()`.

`reporter_for_format` takes `use_color: bool`, **not** `&IoStreams`, to avoid a borrow conflict with the `io.stdout()` writer.

### Errors: domain `Error` + `anyhow::Result`

`src/error.rs` defines a `thiserror`-based `Error` enum. Command handlers return `anyhow::Result<()>` and freely use `.context("…")`. Process exit codes are picked by `exit_code_for_error`, which downcasts the `anyhow::Error` to the domain `Error`:

| Variant | Exit | Notes |
|---|---|---|
| `Error::Config` | 3 | Bad config / missing profile |
| `Error::Auth` | 4 | 401 / 403 from upstream |
| `Error::NotFound` | 2 | 404 from upstream |
| Everything else | 1 | Generic runtime error |

`Error::Template` (P3) and `Error::HttpMiddleware` (P7) both map to exit 1. Master's `Error::InvalidInput` / `Error::InvalidResponse` are typed variants for clap-rejected input and parse failures.

**Rule of thumb**: raise a domain `Error::*` only when you need a specific exit code; otherwise `anyhow::bail!`.

### CLI args: split by service

`src/cli/args/`:
- `mod.rs` — top-level `Cli` struct, `Command` enum, **all global flags** (`-v`, `-q`, `-F`, `-p`, `--config`, `--no-color`, `--no-pager`, `--jq`, `--template`, `--retries`)
- `confluence/{mod,page,space,attachment,blog,comment,content,property,label,admin}.rs` — each subdomain in its own file
- `jira/{mod,issue,board,sprint,project,user,filter,field,workflow,admin}.rs` — same pattern
- `api.rs`, `auth.rs`, `browse.rs`, `alias.rs`, `updater.rs` — single-file commands

Adding a new flag to an existing subcommand: edit the right file under `cli/args/<service>/<subdomain>.rs`. Do **not** add globals on subcommands — they go on `Cli`.

### Command handlers: one entry per service, internal dispatch

`src/cli/commands/`:
- `confluence/{mod,page,space,attachment,blog,comment,content,property,admin}.rs` and `jira/{mod,issue,board,sprint,project,user,filter,field,workflow,admin}.rs` — top-level `mod.rs` is the dispatcher; sub-files implement individual subcommands.
- The top-level `confluence::run` / `jira::run` is where `IoStreams` / `retries` / `transforms` are accepted. Sub-dispatchers return `serde_json::Value` and the top-level `run` calls `write_output` once at the end.
- `api.rs` — generic REST passthrough (`atl api`).
- `auth.rs` — interactive login + status + token + logout (uses `Prompter` and `SecretStore` traits).
- `alias.rs` — `atl alias set/list/delete` + `expand_aliases` (the pre-clap rewriter).
- `browse.rs` — service auto-detect + open in browser.
- `docs.rs` — hidden `atl generate-docs` for man pages / completions / markdown reference.
- `updater.rs` — `atl self check/update`.
- `updater_notifier.rs` — soft "new release" notice printed after success.
- `mod.rs` — `read_body_arg` helper that handles literal / `@file` / `-` (stdin) for `--body` flags.

### HTTP clients: built per call site

`src/client/{mod.rs,confluence.rs,jira.rs}` exposes:
- `build_http_client(instance, retries) -> ClientWithMiddleware` — wraps a `reqwest::Client` in `reqwest_middleware` + `RetryTransientMiddleware` (exponential backoff, 200ms-10s, configurable via `--retries`). Retries apply to **all** methods including POST/PUT/DELETE — users worried about double-submission must pass `--retries 0`.
- `ConfluenceClient::new(&instance, retries)` and `JiraClient::new(&instance, retries)` — typed clients used by the per-service handlers.
- `raw_request(instance, method, endpoint, headers, query, body)` — used by `atl api`. Goes through the same auth + middleware stack.
- `detect_confluence_api_path` — auto-probes `/wiki/rest/api` vs `/rest/api`.

`AtlassianInstance::read_only` causes write methods to be refused at the client layer — respect this in any new code path.

### Auth: env > legacy TOML > keyring

`src/auth/{mod,keyring,prompter}.rs`:
- `SecretStore` trait — `SystemKeyring` (real) and `InMemoryStore` (tests). Backed by the `keyring` crate (macOS Keychain, Windows Credential Manager, Linux Secret Service). Falls back gracefully on headless Linux / Docker by returning `Ok(None)`.
- `Prompter` trait — `InquirePrompter` for production, `MockPrompter` for tests. The `atl auth login` wizard reads through this trait so tests can script interactions.
- `service_name(profile, kind)` — returns `"atl:<profile>:<jira|confluence>"` as the keyring service name.

Token resolution chain (in `AtlassianInstance::resolved_token`):

1. `ATL_API_TOKEN` env var (highest — CI-friendly)
2. Legacy `api_token` field in `atl.toml` (still works; emits a `tracing::warn!` deprecation notice)
3. Keyring lookup
4. None

`atl auth login` writes to the keyring and removes any legacy `api_token` from TOML.

### Config

`src/config/{mod,loader.rs}` + `default_config.toml`. Profiles live at the platform-default config path (`~/.config/atl/atl.toml` on Linux/macOS). `Config` holds `default_profile`, a `HashMap<String, Profile>`, and a top-level `aliases: HashMap<String, String>` table for user aliases.

`ConfigLoader` is an associated-function-style type: `ConfigLoader::load(Option<&Utf8Path>)` and `ConfigLoader::save(&Config, Option<&Utf8Path>)`. `save` preserves existing fields and creates parent directories as needed.

Env overrides: `ATL_CONFIG`, `ATL_PROFILE`, `ATL_API_TOKEN`.

### Body input

`src/cli/commands/mod.rs::read_body_arg` is the canonical handler for `--body`-style flags. It accepts:

- A literal string — `--body "hello"`
- `@path` — read from a file
- `-` — read from stdin

Reuse this rather than reimplementing. Used by Confluence create/update, Jira create/comment, `atl api --input`, etc.

### Aliases pipeline (pre-clap)

`commands::alias::expand_aliases(argv)` runs in `main.rs` before `Cli::parse_from`. Algorithm:

1. Best-effort load of `Config` (silently no-op if it fails — never blocks startup).
2. Walk `argv[1..]` skipping known global flags (and their values for value-taking ones).
3. If the first non-flag token matches a `BUILTINS` constant (which lists every top-level subcommand including `auth`, `browse`, `api`, `alias` itself), return unchanged.
4. Otherwise look up the token in `config.aliases`, split the value with `shlex::split`, and substitute.
5. Run **once** — no recursive expansion, no loop risk.

When you add a new top-level subcommand, also add it to `BUILTINS` in `src/cli/commands/alias.rs` so users can't accidentally shadow it.

### Docs generation

`src/cli/commands/docs.rs` implements a hidden `atl generate-docs --output-dir <DIR>` that walks the clap `Command` tree at runtime and emits:

- `man/atl*.1` via `clap_mangen` (skips clap's synthetic `help` subtree to avoid producing 1900 near-duplicate pages)
- `completions/{atl.bash,_atl,atl.fish,_atl.ps1,atl.elv}` via `clap_complete`
- `reference/atl.md` via a hand-rolled walker

`scripts/gen-docs.sh` builds the debug binary, runs the command, and syncs `docs/reference/`. The CI `docs-up-to-date` job runs this script and `git diff --exit-code` — if you change any user-visible flag, run `scripts/gen-docs.sh` and commit the regenerated `docs/reference/atl.md`.

The release workflow runs the same command against the release binary and packages `dist/_docs/{man,completions}/` into the tarball/zip.

## Branch policy

All changes to `master` **must** go through a pull request — direct pushes are blocked by repository rules. The only exception is the release commit created by `scripts/bump-version.sh`, which is pushed directly to `master` together with the version tag.

## Conventions worth preserving

- **Non-interactive everywhere except `init` and `auth login`.** No spinners, no progress bars, no colour unless `is_stdout_tty()`. Output must be identical under `| cat` and in CI.
- **Never print from a command handler — return a `Value` and let `write_output` do it.** The only exceptions are the `init` wizard and `auth status`/`auth login` console messages, which write through `io.stdout()` directly.
- **Never bypass hooks.** `--no-verify` / `-n` are blocked by `.claude/settings.json` for a reason.
- **Don't widen `Error` casually.** Every new variant changes the exit-code surface. Prefer `.context()` on `anyhow::Error` unless you genuinely need a new exit code.
- **Keep `main.rs` tiny.** Parsing, dispatch, runtime construction. Logic goes in `cli/commands/`.

## Versioning & releases

`Cargo.toml` uses calendar versioning **`YYYY.WW.BUILD`** (e.g. `2026.15.1`). `scripts/bump-version.sh` handles the bump and tag. The release workflow (`.github/workflows/release.yml`) is tag-triggered, builds Linux/macOS/Windows artifacts, and ships `dist/_docs/{man,completions}/` alongside the binary. See `docs/releasing.md` for the operator checklist.

5. One end-to-end command implementation, e.g. `src/cli/commands/jira/issue.rs` + `src/client/jira.rs::get_issue` + `src/cli/args/jira/issue.rs`
